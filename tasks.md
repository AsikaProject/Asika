基于 Asika 当前的能力矩阵和“跨平台 PR 自动化 + 合并队列 + ChatOps”的定位，我认为以下功能能直接提升它在真实生产环境中的适用性，同时不偏离轻量化内核：

---

## 1. 通知/IM 渠道的覆盖面补全（ChatOps）
目前已有 Telegram、Discord、飞书、钉钉、企业微信，但缺少两个高频渠道：
- **Slack**（全球大量团队标准 IM）
- **Microsoft Teams**（企业市场占比较高）

这两个属于 ChatOps 的基础期待，缺失会让很多潜在用户无法落地。

---

## 2. 持久化合并队列（避免重启丢失）
当前合并队列是基于内存的 goroutine 调度。如果 asikad 重启，所有 `waiting` 状态的项会消失，必须等下一个事件重新入队。这对于大规模项目是不可接受的——运维重启可能导致长队重建和合并延迟。

**建议**：将队列状态持久化到 bbolt（结构已有 `queue_items` bucket），启动时恢复。这不会增加资源消耗，但消除了单点故障窗口。

---

## 3. 仓库组级别的权限隔离
目前的用户角色 `admin/operator/viewer` 是全局的。如果被大规模团队使用，应该允许将用户与特定仓库组关联：
- Operator 只能操作被分配的仓库组
- Viewer 只能看自己的仓库组

这是多项目环境下安全合规的基础要求，成本不高，但意义重大（与之前 handler 层的 mode 隔离形成互补）。

---

## 4. 更多的合并策略与自动化
- **线性历史 / Fast-forward only 选项**：很多项目要求线性历史，但目前合并方式只有 merge/squash/rebase，没有强制 fast-forward 检查。
- **自动 rebase**：当 PR 有冲突时，Bot 可以尝试 rebase（配置化，轻量）。
- **/cherry-pick 命令**：通过评论或 ChatOps 命令将特定 PR 的提交 cherry-pick 到指定分支（已支持 Spam 的 cherry-pick，可以泛化为通用能力）。

---

## 5. 统计与 DORA 指标暴露
企业用户会关心：
- 从 PR 打开到合并的周期（Lead Time for Changes）
- 合并频率
- 失败率（Cherry-pick 冲突、CI 失败等）

Asika 已经在记录海量事件，只需暴露一个简单的 `/api/v1/stats` 并利用 `html/template` 在 WebUI 展示仪表板即可。**这是把已有数据转化为高价值可见性的捷径。**

---

## 6. Bitbucket / Codeberg 平台支持
“跨平台”若只有 GitHub、GitLab、Gitea，会漏掉使用 Bitbucket（自托管/云）或 Codeberg 的团队。实现上一个新的平台适配器即可，核心架构不变。

但这一项可以**作为 v2 规划**，不属于紧急缺失。

---

## 7. 集群/主备 模式的基础支持（可选、不推荐）
如果要在大规模环境中真全时可用，可能需要主备切换。这会打破“单二进制 1c1g”的哲学，但可以通过一个简单的 **leader 选举**（基于 bbolt 锁）实现双机热备：一个工作，一个待命。不建议马上做，但可以预留设计。

---

**优先级建议（基于实用度/努力度比）：**

| 优先级 | 功能 |
|--------|------|
| 🔴 高 | 持久化合并队列（重启恢复） |
| 🔴 高 | 仓库级权限隔离 |
| 🟡 中 | Slack 通知渠道 |
| 🟡 中 | DORA 指标仪表板 |
| 🟢 低 | 更多合并策略（Fast-forward 检查） |
| 🟢 低 | Bitbucket 平台适配 |

这些是在保持 Asika 轻量基因的前提下，能让它从“开发者利器”演进到“团队可靠基础设施”的关键补充。
